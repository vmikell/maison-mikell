The highest-risk path, start auth, return to app, persistence, and resume behavior on real phones, has not been validated yet.

What is already done

  • Maison app identity is set to com.maisonmikell.app.
  • Native diagnostics capture app state, pause/resume, launch URL, and appUrlOpen events.
  • Diagnostics history persists through reloads, which matters for redirect auth tracing.
  • Android callback helpers are available through npm run native:android:callback-smoke and npm run native:android:callback-smoke:install.
  • Local verification already passed for build, lint, Capacitor sync, native doctor, and Android debug build generation.

Why this still is not store-ready

  • Google sign-in is still using the Firebase web redirect flow inside the Capacitor shell.
  • Native WebViews do not behave exactly like the hosted site.
  • Return-to-app behavior can still fail even if the web version looks fixed.
  • Store confidence depends on device auth stability more than on more shell polish.

Later today: the first real phone pass

When a phone is available, the goal is to answer one question fast: does Google auth actually survive inside the native shells?

Android sequence

1. Run the readiness checknpm run native:doctor
2. Connect the deviceMake sure adb sees it.
3. Install and callback-smokenpm run native:android:callback-smoke:install
4. Confirm diagnostics evidenceThe panel should log the callback event.
5. Run the real auth passTest Google start, return, persistence, sign-out/sign-back-in, and background/resume.

iPhone sequence

1. Open Xcode on macOSUse ios/App/App.xcodeproj.
2. Configure signingSet up Apple signing for com.maisonmikell.app.
3. Install on a real iPhoneConfirm splash, icon, and clean launch first.
4. Smoke-test the callback pathVerify com.maisonmikell.app://auth returns to Maison.
5. Run the same auth and resume checksUse the diagnostics panel to capture any failure trace.
